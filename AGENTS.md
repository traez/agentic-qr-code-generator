# Agentic QR Code Generator

## Project Overview
A web-based QR code generator with style customization, logo embedding, and export features. Built on Payload CMS 3 + Next.js 16, with PostgreSQL for metadata and Cloudflare R2 for file storage.

## Tech Stack
- **Framework:** Next.js 16 (App Router)
- **CMS/Backend:** Payload CMS 3.85 (with `@payloadcms/next`)
- **Database:** PostgreSQL (via `@payloadcms/db-postgres`)
- **Editor:** Lexical rich text (`@payloadcms/richtext-lexical`)
- **Language:** TypeScript 5.7 (strict)
- **Package Manager:** pnpm
- **Testing:** Vitest (integration) + Playwright (e2e)
- **Linting:** ESLint (Next.js config) + Prettier
- **Infrastructure:** Docker compose (dev), Dockerfile (production)

## Installed Agent Skills
The `.agents/skills/payload/` skill is installed and locked in `skills-lock.json` (sourced from `payloadcms/skills` on GitHub). It provides deep reference material for Payload collections, fields, hooks, access control, queries, endpoints, and more. When working on Payload-related code, consult `.agents/skills/payload/SKILL.md` and the `reference/` docs.

## Key Dependencies
- `qrcode` — QR code generation on Canvas (external dependency)
- Canvas API — Native browser API for rendering
- `sharp` — Image processing for logo embedding/resizing
- `@payloadcms/next` — Payload integration for Next.js

## Architecture & Conventions

### Payload Collections
- Defined in `src/collections/`, exported as `CollectionConfig` objects.
- Registered in `src/payload.config.ts` under `collections: [...]`.
- Use `slug` for collection names (e.g., `qr-codes`, `designs`).
- Access control: function-based — return `true` (full), `false` (deny), or a query constraint for row-level security.
- Hook lifecycle: `beforeValidate`, `beforeChange`, `afterChange`, `beforeRead`, `afterRead`, `beforeDelete`, `afterDelete`.
- Use `versions: { drafts: true }` for draft/publish workflows.
- Use `req.context` in hooks to prevent infinite loops.

### Frontend
- QR generator UI lives in `src/app/(frontend)/`.
- RSC by default; use `"use client"` for interactive QR generation.
- `qrcode` + Canvas API handle generation client-side.
- Styles in `styles.css` (global CSS).

### File Storage
- QR code PNG/SVG + uploaded logos stored in **Cloudflare R2** via Payload upload adapter.
- Metadata (user input, colors, error correction) stored in PostgreSQL via collections.

## Feature Map

| User Story | Implementation | Storage |
|---|---|---|
| **QR Generation** — Enter text/URL, see real-time QR | Client-side `qrcode` lib + Canvas in `(frontend)/page.tsx` | Input text stored in DB collection |
| **Style Customization** — Colors & dot styles | Canvas re-render on state change; hex colors in component state | Design settings stored as JSON field in collection |
| **Logo Embedding** — Upload logo, auto-center | File upload via Payload `Media` collection; Canvas compositing; error correction bumped to M/H | Logo stored in R2 via Media collection |
| **Export & Sharing** — PNG, SVG, copy, share | Canvas `toDataURL`/`toBlob` for PNG; SVG path generation; Clipboard API + Web Share API | Generated files optionally persisted to R2 |

## Important Patterns
1. **QR generation is client-side** — `qrcode` on Canvas. Server stores input data, not images.
2. **Export files (PNG/SVG)** generated client-side, optionally uploaded to R2 via Payload endpoint.
3. **Logo embedding** uses `sharp` server-side or Canvas compositing client-side. Error correction must be M or higher to accommodate logos.
4. **Customization state** (colors, dot style, error level) stored as JSON field in collection for logged-in users; kept client-side for anonymous users.
