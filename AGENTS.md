# Agentic QR Code Generator

## Project Overview
A web-based QR code generator with style customization, logo embedding, save/gallery, and export features. Built on Payload CMS 3 + Next.js 16, with PostgreSQL for metadata and Cloudflare R2 for file storage.

## Tech Stack
- **Framework:** Next.js 16 (App Router)
- **CMS/Backend:** Payload CMS 3.85 (with `@payloadcms/next`)
- **Database:** PostgreSQL (via `@payloadcms/db-postgres`)
- **File Storage:** Cloudflare R2 via `@payloadcms/storage-s3`
- **QR Generation:** `react-qrcode-logo` (wraps `qrcore` internally)
- **Styling:** Tailwind CSS v4 (`@tailwindcss/postcss`)
- **Editor:** Lexical rich text (`@payloadcms/richtext-lexical`)
- **Language:** TypeScript 5.7 (strict)
- **Package Manager:** pnpm
- **Testing:** Vitest (integration) + Playwright (e2e)
- **Linting:** ESLint (Next.js config) + Prettier
- **Infrastructure:** Docker compose (dev), Dockerfile (production)

## Architecture & Conventions

### Payload Collections
- Defined in `src/collections/`, exported as `CollectionConfig` objects.
- Registered in `src/payload.config.ts` under `collections: [...]`.
- Use `slug` for collection names (e.g., `qr-codes`, `media`).
- Access control: function-based — return `true` (full), `false` (deny), or a query constraint for row-level security.
- Hook lifecycle: `beforeValidate`, `beforeChange`, `afterChange`, `beforeRead`, `afterRead`, `beforeDelete`, `afterDelete`.
- Use `req.context` in hooks to prevent infinite loops.

**Users** (`slug: users`)
- Auth enabled (email/password).
- `useAsTitle: 'email'`.
- Minimal fields (email auto-added by Payload).

**Media** (`slug: media`)
- Upload collection storing logos, QR PNGs, QR SVGs.
- Allowed MIME types: `image/png`, `image/svg+xml`, `image/jpeg`, `image/webp`.
- Cache-Control: 1 month.
- Public read/write access (`() => true`).
- Fields: `alt` (text, required), `kind` (select: `logo` / `qr-png` / `qr-svg`).
- Hook: `beforeChange` runs `validateFileName` (lowercase, hyphens, max 50 chars).

**QRCodes** (`slug: qr-codes`)
- Stores each generated QR code record with full style configuration.
- Public read/create access.
- Key fields:
  - `shareId` (text, unique, indexed, auto-generated 8-char UUID)
  - `inputText` (text, required, max 2000 chars)
  - `style` (group): `foregroundColor` (hex), `backgroundColor` (hex), `dotStyle` (squares/dots/fluid)
  - `errorCorrectionLevel` (select: L/M/Q/H, default M)
  - `transparentBackground` (checkbox, default false)
  - `logo` (upload → media, filtered to `kind=logo`)
  - `exportPng` (upload → media, read-only, filtered to `kind=qr-png`)

**Projects** (`slug: projects` — NOT registered in `payload.config.ts`)
- Exists as a collection file but is not included in the Payload config `collections` array.
- Public read access only.
- Fields: `serialNumber` (text), `name` (text, required), `description` (textarea), `metadata` (json).

### Frontend
- QR generator UI lives in `src/app/(frontend)/`, components in `src/components/`.
- `QRCodeDemo.tsx` (client component) — input form + `react-qrcode-logo` <QRCode> for live rendering.
  - Supports text input, foreground/background colors, dot style (squares/dots/fluid), error correction level, logo upload (Base64), size slider.
  - "Generate QR Code" — renders locally; "Generate & Save" — calls `saveQRCode` Server Action.
  - Logo upload auto-bumps error correction to M if currently L.
- `QRCodeGallery.tsx` (client component) — fetches saved QRs via `listQRCodes` Server Action, displays grid with detail modal.
  - Detail modal: download PNG (`downloadQRCode`), copy PNG to clipboard (`generateBlob` + `ClipboardItem`), shareable link (`#qr-{shareId}`).
  - Hash-based routing: `/#qr-{shareId}` auto-opens matching QR on load.
- Styles in `styles.css` (Tailwind v4 CSS with `@theme` custom properties + skeleton animation).

### Server Actions
- `src/actions/qrcodes/saveQRCode.ts` — Accepts input text, style config, optional logo (Base64). If logo provided, creates Media record (file + metadata) first, then creates QRCodes record with logo relation.
- `src/actions/qrcodes/listQRCodes.ts` — Returns all QRCodes sorted by `-createdAt` with depth 1 (includes related Media data).

### File Storage
- QR code logos + export files stored in **Cloudflare R2** via `@payloadcms/storage-s3` plugin.
- The `media` collection has S3 storage enabled; files are uploaded directly to R2.
- Metadata (user input, colors, error correction, relations) stored in PostgreSQL via collections.

## Feature Map

| User Story | Implementation | Storage |
|---|---|---|
| **QR Generation** — Enter text/URL, see real-time QR | `react-qrcode-logo` `<QRCode>` component in `QRCodeDemo.tsx` | Input text + style saved to `QRCodes` collection |
| **Style Customization** — Colors & dot styles | State-driven re-render of `<QRCode>`; hex colors + dotStyle + EC level in component state | Style stored as `QRCodes.style` group field (JSON) |
| **Logo Embedding** — Upload logo, auto-center | Local file → Base64 → `<QRCode logoImage>` prop; error correction auto-bumped to M | Logo optionally uploaded to `Media` collection → R2 during save |
| **Save & Gallery** — Persist QR codes to DB | Server Action `saveQRCode` creates QRCodes doc (with optional Media relation); `listQRCodes` fetches all | QRCodes records in PostgreSQL; logos in R2 via Media |
| **Export PNG** — Download as PNG | `react-qrcode-logo`'s `downloadQRCode()` API | — |
| **Copy to Clipboard** — Copy QR as image | `generateBlob()` + `navigator.clipboard.write()` with `ClipboardItem` | — |
| **Share via Link** — Hash-based sharing | URL hash `#qr-{shareId}` auto-opens matching QR in gallery on page load | `shareId` in QRCodes collection |

## Important Patterns
1. **QR generation is client-side** — `react-qrcode-logo` renders via Canvas in the browser. Server stores input data and style config, not images.
2. **Data flow**: Client form → Server Action (`saveQRCode`) → Payload API → PostgreSQL (QRCodes record) + optionally R2 (Media file).
3. **Logo embedding** is client-side via `react-qrcode-logo`'s built-in logo image overlay. Error correction must be M or higher for reliable scanning. `sharp` is only used by Payload internally for thumbnail generation, not for custom logo processing.
4. **Customization state** (colors, dot style, error level) stored as JSON group field in QRCodes collection for saved records; kept in component state during editing.
5. **Server Actions** are the bridge between client and Payload. They use `getPayload({ config })` directly, not REST endpoints.
6. **Share links** use URL hash fragments (`/#qr-{shareId}`) — no server-side routing needed. The `QRCodeGallery` component reads `window.location.hash` on mount.
